# Stage 1: PRD（产品需求文档）

## 概述

| 项目 | 说明 |
|------|------|
| **执行者** | PM（产品经理） |
| **产出物** | 《产品需求文档》 |
| **下游依赖** | Stage 2 DDD Design |
| **闸口条件** | 需求清晰、边界明确、验收标准可测 |

## 职责边界

**PM 负责**：
- 定义"做什么"和"为什么做"
- 明确功能边界和优先级
- 定义验收标准

**PM 不负责**：
- 技术实现方案
- 领域模型设计
- 接口定义

## PRD 核心要素

### 1. 背景与目标

```markdown
## 背景
- 业务背景：为什么要做这个功能
- 用户痛点：解决什么问题
- 业务价值：带来什么收益

## 目标
- 核心目标：一句话描述
- 量化指标：可衡量的成功标准
```

### 2. 用户故事

```markdown
## 用户故事

作为 [角色]，
我想要 [功能]，
以便 [价值]。

### 示例
作为会员用户，
我想要每天自动获得 100 点券，
以便用于商城消费抵扣。
```

### 3. 功能范围

```markdown
## 功能范围

### 包含（In Scope）
- 功能点 1
- 功能点 2

### 不包含（Out of Scope）
- 明确排除的功能
- 后续迭代的功能

### 依赖
- 依赖的外部系统
- 依赖的前置功能
```

### 4. 验收标准

```markdown
## 验收标准

### AC-1: [验收项名称]
- Given: 前置条件
- When: 用户操作
- Then: 期望结果

### AC-2: [验收项名称]
- Given: ...
- When: ...
- Then: ...
```

## 闸口检查清单

进入 Stage 2 前，必须确认：

- [ ] 背景和目标清晰，能回答"为什么做"
- [ ] 用户故事完整，能回答"给谁用"
- [ ] 功能范围明确，有清晰的 In/Out Scope
- [ ] 验收标准可测，能写成测试用例
- [ ] 无技术实现细节（那是 Stage 2 的事）

## 常见问题

### Q1: PRD 需要写到多细？

**原则**：写到能让架构师理解业务意图即可。

```
❌ 太粗：用户可以订阅会员
✅ 合适：用户可以订阅月度/年度会员，订阅后每天获得 100 点券
❌ 太细：点击订阅按钮后调用 /api/membership/subscribe 接口
```

### Q2: 验收标准和测试用例的区别？

- **验收标准**：业务视角，描述"什么是正确的"
- **测试用例**：技术视角，描述"如何验证正确"

验收标准在 PRD 中定义，测试用例在 Stage 3 规格建模中推导。

### Q3: 需求变更怎么处理？

1. 更新 PRD 文档
2. 评估对下游的影响（DDD 设计、规格建模、工件推导）
3. 触发受影响阶段的重新 Review
